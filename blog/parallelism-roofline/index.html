Parallelism by the Roofline

May 10, 2026

A training step is a fight over three budgets: memory per device, compute per device, and communication between devices. Every parallelism strategy is a different way of paying those budgets for the same arithmetic.

To keep the strategies comparable, we use a single example. The whole post comes back to one linear layer:

X[B,E]  @  W[E,E]→Y[B,E]X[B, E] \; @ \; W[E, E] \rightarrow Y[B, E]X[B,E]@W[E,E]→Y[B,E]

The cost is roughly 2BE22 B E^22BE2 floating point operations. That is the train we want to keep moving. The question every strategy answers in its own way:

What is split, what is replicated, what communication repairs the split, and can that communication be hidden under matmul compute?

The “communication” piece is always one of a small set of collectives: all-gather, reduce-scatter, all-reduce, and (occasionally) all-to-all. Each is defined and animated in the appendix — every mention below links to its panel.

Data Parallelism

Across chips, every WWW is identical; each chip owns a different chunk of the batch dimension. The forward pass is literally just parallelized. The repair step comes at the end: the gradients of the replicated WWW must agree across devices, so an all-reduce of ∂W\partial W∂W closes the loop.

The win is throughput, not memory. Every device still stores the full model, the full gradient, and the full optimizer state.

Where, then, is the compute-vs-comm overlap? The forward pass has none — every chip just multiplies locally. The overlap lives in the backward pass: as each layer’s gradients become available, their all-reduce can run while earlier layers are still computing their backward.

One fact about that all-reduce, worth carrying into the next section: an all-reduce is exactly a reduce-scatter followed by an all-gather — the appendix’s all-reduce panel literally pauses on the reduce-scatter result mid-step. The bandwidth-optimal ring algorithm is implemented as those two phases — each moves (N−1)M/N(N{-}1)M/N(N−1)M/N bytes per chip, summing to the all-reduce’s familiar 2(N−1)M/N2(N{-}1)M/N2(N−1)M/N. FSDP is what happens when you take that all-reduce and pull it apart in time: an all-gather of WWW in forward, a reduce-scatter of ∂W\partial W∂W in backward. Same total volume as DP, very different memory profile.

FSDP

Stop replicating. The batch is still sharded as in DP, but now WWW is sliced too — each chip stores only a strip of WWW‘s rows. The mental model that makes this work: at any moment, only one or two layers are materialized (yellow in the picture above). The other 50 or 100 layers stay sharded. Memory shrinks by the shard count.

The performance question is whether the gather for layer N+1N{+}1N+1 can finish while layer NNN is still computing. Toggle the modes below to see the difference: in the overlapped schedule the compute stream is continuous; in the naive schedule each matmul waits for its all-gather to land, and the step takes roughly twice as long.

The same per-layer all-gather will return in the backward pass — computing ∂X\partial X∂X also needs the full WWW — so backward either repeats the gather or reuses a forward-cached copy. We’ll see that mirroring later.

Tensor Parallelism

A different shape. Tensor parallelism splits the matrix itself. The cleanest mental model is row-parallel: there is one shared XXX, but each chip holds a slice of its columns (along the contraction dimension EEE), paired with the matching rows of WWW. Each device then computes

Xi [B,E/P]  @  Wi [E/P,E]→Y(i) [B,E]X_i \, [B, E/P] \; @ \; W_i \, [E/P, E] \rightarrow Y^{(i)} \, [B, E]Xi​[B,E/P]@Wi​[E/P,E]→Y(i)[B,E]

Notice the shape: Y(i)Y^{(i)}Y(i) is the full output, not a slice. But it is a partial sum — only the iii-th of PPP terms in the contraction. None of the chips alone has the correct answer. The picture below shows it: every chip’s YYY block is colored across its entire shape (striped, to mark “partial”), because every chip has an opinion about every output element, and they all have to be added together.

That is exactly the collective the matrix shape demands: a sum across devices that re-shards the output. Press Reduce-scatter Y above — the striped partials disappear and each chip is left with one column of the final YYY. That collective is the entire reason tensor parallelism communicates inside the layer.

We cannot overlap between layers here — the next layer’s input is exactly the current layer’s reduce-scattered output, so there is nothing yet to communicate ahead of. The overlap happens inside the layer, with the matmul work split into a staircase that interleaves partial computation with a reduce-scatter on the previous partial.

The useful ratio is E/PE / PE/P: compute scales like BE2/PB E^2 / PBE2/P and the activation collective is closer to BEB EBE, so larger EEE buys room to hide communication and larger PPP makes each local matmul thinner.

FSDP + TP

FSDP and TP are not alternatives — they are orthogonal axes of the same problem. FSDP shrinks memory per device by sharding the model state. TP shrinks compute per device by slicing each layer’s matmul. The first attacks weight memory; the second attacks per-chip compute. Together, on a 2D mesh, you get both.

This matters most when the per-chip batch gets small. Pure FSDP needs each chip to do enough work for the all-gather to hide under the matmul — empirically that’s around 850 tokens per chip on a TPU pod. Pure TP needs the local matmul to stay above the bandwidth knee, which limits how thin you can slice it. Stack them and the JAX scaling book reports a per-chip floor closer to ~100 tokens — roughly an 8× shrink.

The mesh is 2D: one axis carries FSDP, the other carries TP. The same matmul as before,

X[B,E]  @  W[E,E]→Y[B,E]X[B, E] \; @ \; W[E, E] \rightarrow Y[B, E]X[B,E]@W[E,E]→Y[B,E]

is sharded twice: XXX‘s batch is split along the FSDP axis, XXX‘s contraction is split along the TP axis, and WWW‘s rows are split along both axes — each chip owns a tiny block.

Forward, per layer, has two collectives — one on each axis:

  1. AG along the FSDP axis: chips with the same TP rank pool their FSDP shards of WWW. After this, each chip holds the W slice it needs for its local matmul — its TP slice of the contraction rows, no longer further split across FSDP.
  2. Local matmul. No comm needed: each chip multiplies its X slice by its W slice, summing over its TP slice of the contraction. The result is a partial-sum output at full shape — every chip has an opinion on every column of its batch slice, but the sum across TP is incomplete.
  3. RS along the TP axis: chips with the same FSDP rank sum-and-scatter the partials. Each chip ends up owning one TP slice of the final YYY.

The output’s layout matches XXX‘s, so the next layer can start without an intermediate re-shard.

Now look at how the three streams interleave once the pipeline is moving. The next layer’s AG can begin on the FSDP axis while the current layer’s matmul and TP-axis RS are still running — different axes, no contention. The diagram below is the steady state, mid-pipeline: L1’s W is already gathered, its matmul is being chunked and reduce-scattered along TP, and L2’s W is prefetching along FSDP.

The key shift from pure FSDP or pure TP: both axes carry per-layer traffic, but with different collectives. AG on the FSDP axis, RS on the TP axis. The roofline question splits in two — one per axis — and the optimal mesh shape balances them. Bigger batches push the FSDP axis larger; the bandwidth ratio between the two axes pulls the other way. The scaling book has the closed-form rule.

The Backward Pass

Forward is only half the step. Backward needs gradients, and getting the gradients into the right shape — sharded, replicated, summed — is where each strategy spends its second round of communication.

The mental flip is simple. Wherever the forward did an all-gather, backward will reduce-scatter; wherever the forward did a reduce-scatter, backward will all-gather. The same overlap rules apply. But the frequencies and axes change strategy by strategy.

DP — One all-reduce at the end

W is replicated on every chip, so every chip has computed its own gradient ∂W(i)\partial W^{(i)}∂W(i) from its batch slice. To make the gradients agree (so the optimizer steps the replicated parameters identically), DP fires one all-reduce of ∂W\partial W∂W at the end of the step.

FSDP — One reduce-scatter per layer

Backward visits the layers in reverse. For each layer, the full WWW is gathered (we need it to compute ∂X\partial X∂X and ∂W\partial W∂W), the gradient is computed at full shape, then reduce-scatter sends each chip back to owning just its row of ∂W\partial W∂W. The mirror of forward’s gather pipeline.

TP — Within-layer dual staircase

Forward’s row-parallel TP needed a reduce-scatter inside each layer (after the partial-sum matmul). Backward needs the dual: an all-gather inside each layer to broadcast the upstream ∂Y\partial Y∂Y back to all chips before each one computes its local ∂W\partial W∂W shard. Same staircase shape, mirrored.

FSDP + TP — collectives swapped

The 2D backward mirrors the forward, with both collectives swapped across axes. Forward fired AG along the FSDP axis (to gather WWW) and RS along the TP axis (to reduce-scatter the partial-sum YYY). Backward fires the other collective on each axis: AG along the TP axis (broadcasting the upstream ∂Y\partial Y∂Y) and RS along the FSDP axis (reducing the full-shape ∂W\partial W∂W back into the FSDP layout). Same per-layer cost on each axis, opposite collective.

The Unified Picture

The strategies differ less by vocabulary than by repair step.

StrategyWhat is split?What is replicated?Main repairFailure mode
Data Parallelismbatch rowsweights, gradients, optimizer stategradient syncmodel state does not fit
FSDPbatch rows + model stateone layer’s weights, transientlyall-gather, reduce-scatterexposed all-gather stalls
Tensor ParallelismXXX’s contraction + WWW‘s rowsnothing; each chip’s output is a partial sumreduce-scatter YYY inside the layerlocal matmuls too small
FSDP + TPboth axes of a 2D mesh: batch + model state along FSDP, layer compute along TPnothingAG along FSDP + RS along TP, every layerwrong mesh shape — unbalanced axes

Matmul is the train. Collectives lay down the track. If the track arrives before the train needs it, the schedule looks compute-bound. If the track arrives late, the roofline drops to the network roof.
