A training step is a fight over three budgets: memory per device, compute per device, and communication between devices. Every parallelism strategy is a different way of paying those budgets for the same arithmetic.

To keep the strategies comparable, we use a single example. The whole post comes back to one linear layer:

$$
X[B, E] \; @ \; W[E, E] \rightarrow Y[B, E]
$$

The cost is roughly $2 B E^2$ floating point operations. That is the train we want to keep moving. The question every strategy answers in its own way:

> What is split, what is replicated, what communication repairs the split, and can that communication be hidden under matmul compute?

The "communication" piece is always one of a small set of collectives: **all-gather**, **reduce-scatter**, **all-reduce**, and (occasionally) **all-to-all**. Each is defined and animated in the [appendix](#appendix-the-collectives) — every mention below links to its panel.

# Data Parallelism

Replicate the weights, split the batch. Each device sees a different slice of $X$'s rows, multiplies by the same $W$, and produces its slice of $Y$:

<ShardingStage strategy="dp" client:visible />

Across chips, every $W$ is identical; each chip owns a different chunk of the batch dimension. The forward pass is literally just parallelized. The repair step comes at the end: the gradients of the replicated $W$ must agree across devices, so an [all-reduce](#collective-all-reduce) of $\partial W$ closes the loop.

The win is throughput, not memory. Every device still stores the full model, the full gradient, and the full optimizer state.

Where, then, is the compute-vs-comm overlap? The forward pass has none — every chip just multiplies locally. The overlap lives in the *backward* pass: as each layer's gradients become available, their all-reduce can run while earlier layers are still computing their backward.

<OverlapTimeline strategy="dp" client:visible />

One fact about that all-reduce, worth carrying into the next section: **an [all-reduce](#collective-all-reduce) is exactly a [reduce-scatter](#collective-reduce-scatter) followed by an [all-gather](#collective-all-gather)** — the appendix's all-reduce panel literally pauses on the reduce-scatter result mid-step. The bandwidth-optimal ring algorithm is implemented as those two phases — each moves $(N{-}1)M/N$ bytes per chip, summing to the all-reduce's familiar $2(N{-}1)M/N$. FSDP is what happens when you take that all-reduce and pull it apart in time: an all-gather of $W$ in forward, a reduce-scatter of $\partial W$ in backward. Same total volume as DP, very different memory profile.

# FSDP

Stop replicating. The batch is still sharded as in DP, but now $W$ is sliced too — each chip stores only a strip of $W$'s rows. In the picture below, $Y$ starts empty: chip $i$ has its $X$ slice and its $W$ shard, but it cannot compute its $Y$ yet — the matmul $X_i @ W_i$ doesn't even compose. Press *All-gather W* to materialize the full $W$ on every chip; only then can each chip produce its batch slice of $Y$.

<ShardingStage strategy="fsdp" client:visible />

The mental model that makes this work: at any moment, only one or two layers are *materialized* (yellow in the picture above). The other 50 or 100 layers stay sharded. Memory shrinks by the shard count.

The performance question is whether the gather for layer $N{+}1$ can finish while layer $N$ is still computing. Toggle the modes below to see the difference: in the overlapped schedule the compute stream is continuous; in the naive schedule each matmul waits for its all-gather to land, and the step takes roughly twice as long.

<OverlapTimeline strategy="fsdp" client:visible />

The same per-layer all-gather will return in the backward pass — computing $\partial X$ also needs the full $W$ — so backward either repeats the gather or reuses a forward-cached copy. We'll see that mirroring later.

# Tensor Parallelism

A different shape. Tensor parallelism splits the matrix itself. The cleanest mental model is *row-parallel*: there is one shared $X$, but each chip holds a slice of its columns (along the contraction dimension $E$), paired with the matching rows of $W$. Each device then computes

$$
X_i \, [B, E/P] \; @ \; W_i \, [E/P, E] \rightarrow Y^{(i)} \, [B, E]
$$

Notice the shape: $Y^{(i)}$ is the *full* output, not a slice. But it is a **partial sum** — only the $i$-th of $P$ terms in the contraction. None of the chips alone has the correct answer. The picture below shows it: every chip's $Y$ block is colored across its entire shape (striped, to mark "partial"), because every chip has an opinion about every output element, and they all have to be added together.

<ShardingStage strategy="tp" client:visible />

That is exactly the collective the matrix shape demands: a sum across devices that re-shards the output. Press *Reduce-scatter Y* above — the striped partials disappear and each chip is left with one column of the final $Y$. That collective is the entire reason tensor parallelism communicates inside the layer.

We cannot overlap between layers here — the next layer's input is exactly the current layer's reduce-scattered output, so there is nothing yet to communicate ahead of. The overlap happens *inside* the layer, with the matmul work split into a staircase that interleaves partial computation with a reduce-scatter on the previous partial.

<TpStaircase client:visible />

The useful ratio is $E / P$: compute scales like $B E^2 / P$ and the activation collective is closer to $B E$, so larger $E$ buys room to hide communication and larger $P$ makes each local matmul thinner.

# FSDP + TP

FSDP and TP are not alternatives — they are orthogonal axes of the same problem. FSDP shrinks **memory per device** by sharding the model state. TP shrinks **compute per device** by slicing each layer's matmul. The first attacks weight memory; the second attacks per-chip compute. Together, on a 2D mesh, you get both.

This matters most when the per-chip batch gets small. Pure FSDP needs each chip to do enough work for the all-gather to hide under the matmul — empirically that's around 850 tokens per chip on a TPU pod. Pure TP needs the local matmul to stay above the bandwidth knee, which limits how thin you can slice it. Stack them and the [JAX scaling book](https://jax-ml.github.io/scaling-book/training/) reports a per-chip floor closer to ~100 tokens — roughly an 8× shrink.

The mesh is 2D: one axis carries FSDP, the other carries TP. The same matmul as before,

$$
X[B, E] \; @ \; W[E, E] \rightarrow Y[B, E]
$$

is sharded twice: $X$'s batch is split along the FSDP axis, $X$'s contraction is split along the TP axis, and $W$'s rows are split along **both** axes — each chip owns a tiny block.

<ShardingStage strategy="fsdp_tp" numChips={4} E={4} client:visible />

Forward, per layer, has two collectives — one on each axis:

1. **AG along the FSDP axis:** chips with the same TP rank pool their FSDP shards of $W$. After this, each chip holds the W slice it needs for its local matmul — its TP slice of the contraction rows, no longer further split across FSDP.
2. Local matmul. No comm needed: each chip multiplies its X slice by its W slice, summing over its TP slice of the contraction. The result is a **partial-sum** output at full shape — every chip has an opinion on every column of its batch slice, but the sum across TP is incomplete.
3. **RS along the TP axis:** chips with the same FSDP rank sum-and-scatter the partials. Each chip ends up owning one TP slice of the final $Y$.

The output's layout matches $X$'s, so the next layer can start without an intermediate re-shard.

Now look at how the three streams interleave once the pipeline is moving. The next layer's AG can begin on the FSDP axis while the current layer's matmul and TP-axis RS are still running — different axes, no contention. The diagram below is the steady state, mid-pipeline: L1's W is already gathered, its matmul is being chunked and reduce-scattered along TP, and L2's W is prefetching along FSDP.

<FsdpTpStaircase client:visible />

The key shift from pure FSDP or pure TP: **both axes carry per-layer traffic**, but with different collectives. AG on the FSDP axis, RS on the TP axis. The roofline question splits in two — one per axis — and the optimal mesh shape balances them. Bigger batches push the FSDP axis larger; the bandwidth ratio between the two axes pulls the other way. The [scaling book](https://jax-ml.github.io/scaling-book/training/) has the closed-form rule.

# The Backward Pass

Forward is only half the step. Backward needs gradients, and getting the gradients into the right shape — sharded, replicated, summed — is where each strategy spends its second round of communication.

The mental flip is simple. Wherever the forward did an all-gather, backward will reduce-scatter; wherever the forward did a reduce-scatter, backward will all-gather. The same overlap rules apply. But the *frequencies* and *axes* change strategy by strategy.

## DP — One all-reduce at the end

W is replicated on every chip, so every chip has computed its own gradient $\partial W^{(i)}$ from its batch slice. To make the gradients agree (so the optimizer steps the replicated parameters identically), DP fires **one all-reduce of $\partial W$** at the end of the step.

<DpBackward client:visible />

## FSDP — One reduce-scatter per layer

Backward visits the layers in reverse. For each layer, the full $W$ is gathered (we need it to compute $\partial X$ and $\partial W$), the gradient is computed at full shape, then reduce-scatter sends each chip back to owning just its row of $\partial W$. The mirror of forward's gather pipeline.

<FsdpBackward client:visible />

## TP — Within-layer dual staircase

Forward's row-parallel TP needed a reduce-scatter inside each layer (after the partial-sum matmul). Backward needs the dual: an **all-gather** inside each layer to broadcast the upstream $\partial Y$ back to all chips before each one computes its local $\partial W$ shard. Same staircase shape, mirrored.

<TpBackward client:visible />

## FSDP + TP — collectives swapped

The 2D backward mirrors the forward, with both collectives swapped across axes. Forward fired AG along the FSDP axis (to gather $W$) and RS along the TP axis (to reduce-scatter the partial-sum $Y$). Backward fires the other collective on each axis: AG along the TP axis (broadcasting the upstream $\partial Y$) and RS along the FSDP axis (reducing the full-shape $\partial W$ back into the FSDP layout). Same per-layer cost on each axis, opposite collective.

# The Unified Picture

{/* TODO: <UnifiedRoofline /> — all three roofs + four strategies plotted as points; global sliders */}

The strategies differ less by vocabulary than by repair step.

| Strategy | What is split? | What is replicated? | Main repair | Failure mode |
| --- | --- | --- | --- | --- |
| Data Parallelism | batch rows | weights, gradients, optimizer state | gradient sync | model state does not fit |
| FSDP | batch rows + model state | one layer's weights, transiently | all-gather, reduce-scatter | exposed all-gather stalls |
| Tensor Parallelism | $X$'s contraction + $W$'s rows | nothing; each chip's output is a partial sum | reduce-scatter $Y$ inside the layer | local matmuls too small |
| FSDP + TP | both axes of a 2D mesh: batch + model state along FSDP, layer compute along TP | nothing | AG along FSDP + RS along TP, every layer | wrong mesh shape — unbalanced axes |

Matmul is the train. Collectives lay down the track. If the track arrives before the train needs it, the schedule looks compute-bound. If the track arrives late, the roofline drops to the network roof.
